Once InQL is installed, you can scan a GraphQL endpoint and start exploring its schema in a few steps. This guide walks you through opening the Scanner, running your first introspection scan, and sending generated queries to other Burp tools. You should be up and running in under five minutes.

## What introspection does

GraphQL servers expose a built-in mechanism called **introspection** that lets clients query the API's own schema — every type, field, query, mutation, and subscription the server supports. InQL sends an introspection query to your target endpoint and uses the response to automatically generate every possible operation, organized for you to browse, test, and send to Burp tools.

If the target has introspection disabled, see the [Schema Bruteforcer](/features/schema-bruteforcer) feature, which can recover schema details without it.

## Scan your first endpoint

<Steps>
  <Step title="Open the InQL tab">
    In Burp Suite, click the **InQL** tab in the main toolbar. You'll see three sub-tabs: **Scanner**, **Batch Queries**, and **Engine Fingerprinting**. Click **Scanner**.
  </Step>
  <Step title="Enter the endpoint URL">
    In the Scanner's URL field, type or paste the full URL of the GraphQL endpoint you want to test. For example:

    ```text
    https://example.com/graphql
    ```

    If the endpoint requires authentication, set your headers before scanning. InQL lets you configure per-domain custom headers — look for the headers section in the Scanner tab.
  </Step>
  <Step title="Run the scan">
    Click **Analyze**. InQL sends an introspection query to the endpoint and processes the response. Depending on schema size, this usually takes a few seconds.

    When the scan completes, the left panel populates with a tree of all queries, mutations, and subscriptions InQL discovered.
  </Step>
  <Step title="Browse the generated queries">
    Click any item in the tree to view the auto-generated query in the right panel. InQL generates valid GraphQL with proper nesting based on the schema, up to the configured depth (default: 2 levels).

    The output panel shows the query formatted and ready to use.
  </Step>
  <Step title="Run a Points of Interest scan">
    Click **Scan for Points of Interest** to analyze the schema for fields that may be security-relevant. InQL categorizes findings across several categories:

    - **Auth** — authentication and authorization fields
    - **PII** — fields likely to contain personal data
    - **Payment** — payment and billing operations
    - **Files** — file upload or download operations
    - **Debugging** — fields that may expose internal state
    - **Deprecated** — operations marked as deprecated

    Results appear as a separate view. You can enable or disable categories to focus on what matters for your engagement.
  </Step>
  <Step title="Check for circular references">
    After scanning, InQL automatically identifies circular reference patterns in the schema — recursive types that could be exploited to craft deeply nested queries for denial-of-service attacks. Any detected circular references appear flagged in the scanner results.
  </Step>
  <Step title="Send queries to Burp tools">
    Right-click any generated query in the Scanner and choose **Send to Repeater** or **Send to Intruder** to pass it directly into those tools. The request is pre-filled with the correct content type and the generated query body, ready for you to modify and test.
  </Step>
</Steps>

## Using InQL in the message editor

InQL registers a **GraphQL (InQL)** tab on every HTTP message editor in Burp — Proxy, Repeater, Intruder, and others. When you intercept or open a GraphQL request, click the **GraphQL (InQL)** tab to see the request body formatted with syntax highlighting.

You can edit queries directly in this view and send the modified request back to any Burp tool.
